\section{Training pipeline}
\label{sec:training-pipeline}

Meowtion learns new behaviours through a closed loop that an owner drives entirely from the
dashboard: capture, label, train, deliver, run. No local toolchain is required.

\subsection{The loop}

\begin{enumerate}[leftmargin=1.5em]
  \item \textbf{Define behaviours.} The owner lists the actions to recognise (eat, drink, purr,
        scratch, litter-tray, and so on) in the dashboard. This set drives both the labelling UI
        and the trained model's outputs.
  \item \textbf{Capture.} With a station in training mode, short audio clips and time-aligned
        motion are recorded while a registered collar is in range, and uploaded to Cloud Storage.
  \item \textbf{Label.} The owner reviews clips in the dashboard, sets each clip's action, trims
        dead space, and discards poor examples.
  \item \textbf{Train.} A Cloud Function trains the model(s) server-side from the labelled clips
        and writes the results back: the quantised \texttt{.tflite} blobs to Storage and the
        version and label list to the database.
  \item \textbf{Deliver.} Each station watches a model-version key. When it changes, the station
        downloads the new \texttt{.tflite} files and pushes them to the collar over the air
        (\cref{sec:protocols}).
  \item \textbf{Run.} The owner switches the station to production mode; the collar classifies
        with the new model and the dashboard shows live, named behaviours.
\end{enumerate}

\subsection{Label-agnostic by design}

The trainer is not hard-coded to a fixed set of actions. It learns whatever label set the owner
defines, which is what makes Meowtion a platform rather than a fixed-function gadget. The same
pipeline that recognises eating and drinking can be pointed at any distinguishable behaviour the
owner can capture and label.

\subsection{Delivery contract}

Training outputs are written to predictable locations so the station can find them without
coordination: the model blobs land in Storage under the owner's path as
\texttt{imu\_model.tflite} and \texttt{audio\_model.tflite}, and the new version is mirrored into
each of that owner's stations' configuration so the station's gate fires. The label list is
stored alongside so the dashboard can map a class index back to a name (\cref{sec:cloud-backend}).

\subsection{Known limitation}

A two-class model (for example, eat versus drink only) has no way to represent ``neither'' and
will confidently assign one of its two labels to an idle cat. The fix is to retrain with an
explicit idle/other class; this is a data and modelling task, not a firmware change, and is
tracked in \cref{sec:future-work}.
