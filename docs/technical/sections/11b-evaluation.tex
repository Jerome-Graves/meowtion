classification accuracy of each cascade stage, the benefit of the audio confirmation stage, the
on-device resource footprint and inference latency, power draw and battery life, and the time taken to
deliver a model over the air. All on-device figures are measured on the collar hardware
(\cref{sec:hardware}); accuracy figures are reported on a held-out test split of the labelled clips.

\subsection{Power and battery life}

Power is the defining constraint on the wearable (NFR-1). The dominant lever is the rules-based
activity gate (\cref{sec:on-device-ai,sec:firmware-collar}): a resting cat puts the collar into
low-power rest, where the 104\,Hz sampling and inference stop, the IMU drops to a low-power watch, and
the radio slows, so average current tracks how active the cat actually is instead of running flat out
all day. The remaining draw is set by the active duty cycle of the IMU path and the occasional audio
path on the 100\,mAh cell.

\subsection{Testing and verification}

The system's logic is covered by automated unit tests that run in continuous integration on every
change, and the integration and on-device behaviour is verified on the hardware. The security
controls and their evidence are detailed separately (\cref{sec:security-privacy}).

\begin{itemize}[leftmargin=1.5em]
  \item \textbf{Automated unit tests.} 80 Python tests cover the dashboard data layer (event
        aggregation, label resolution, the low-power-rest event handling, the time-window
        drill-down, the per-day timeline segmenting, and the health comparison) and the
        Cloud Function's signal logic (per-window normalisation, windowing, WAV parsing, and the
        path-validation regexes), including property and fuzz tests that hammer the untrusted-input
        surface (the path validators and the WAV parser) with adversarial and seeded-random inputs.
        Thirteen host-compiled C tests cover the on-device cascade (the rules-based activity gate and
        the confidence-gated IMU/audio gating) and the $\mu$-law audio codec and model-input
        representation, the latter swept exhaustively over the whole int16 domain and checked to match
        the training representation. The full per-test inventory and results are listed in
        \cref{app:tests}.
  \item \textbf{Coverage.} Line coverage of the unit-testable core is 100\,\%. The Firebase request
        handlers, model training, the dashboard UI, and the hardware-bound firmware paths are
        excluded from that figure and are covered by the on-hardware checks below.
  \item \textbf{Continuous integration.} Every push and pull request runs the Python and C test
        suites and CodeQL static security analysis.
  \item \textbf{On-hardware verification.} Model load-from-flash, on-device classification, the
        activity gate's rest/wake cycle, the rest-event logging, and the build and backend-deploy
        steps were each confirmed directly on the collar and station.
\end{itemize}

\subsection{Limitations}

The current evaluation is single-cat and within-subject; generalisation across cats and households is
addressed as future work (\cref{sec:future-work}). The figures above are for the behaviour set used in
testing; accuracy on a larger or more fine-grained label set should be re-measured as the set grows.
